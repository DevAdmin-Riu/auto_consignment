# 에러 핸들링 감사 보고서

> 각 벤더 주문 프로세스의 에러 처리 현황과 수정 계획

## 공통 프로세스

```
1. 로그인
2. 장바구니 비우기
3. 상품별: URL 이동 → 옵션 선택 → 수량 입력 → 가격 확인 → 장바구니 담기
4. 장바구니 검증 (배치만)
5. 배송지 입력
6. 결제
7. 주문번호 확인
8. 결과 저장 (saveOrderResults → 대행접수 → 출고)
9. 결제 로그 저장 (createPaymentLogs)
```

| N-2 | 상품 처리 실패 (배치 중 일부) | continue → 나머지로 계속 | 3개 중 1개 실패해도 2개로 결제 → 수량 틀어짐 | 🔴 CRITICAL | ❌ 미수정 |
| N-3 | 주문하기 버튼 못 찾음 | success: false 기록만, 계속 진행 | 결제 페이지 안 넘어가는데 계속 진행 | 🔴 CRITICAL | ❌ 미수정 |
| N-4 | 주문번호 못 찾음 | success: true로 반환 | 결제 됐는데 주문번호 미기록 → 수동 확인 불가 | 🟡 HIGH | ❌ 미수정 |
| N-5 | 결제 로그 저장 실패 | "(무시)" catch | 결제 금액 추적 안 됨 | 🟡 HIGH | ❌ 미수정 |

### N-1 수정 방향
```
장바구니 비우기 실패 → saveOrderResults(false) + return
이유: 이전 주문 잔여 상품이 같이 결제될 위험
```

### N-2 수정 방향
```
배치 주문에서 상품 1개라도 장바구니 담기 실패 시 → 전체 그룹 중단
이유: 주문 수량이 틀어지면 결제 금액도 틀어짐
예외: 담당자 확인 필요(needsManagerVerification)인 경우만 스킵 허용
```

### N-3 수정 방향
```
주문하기 버튼 못 찾음 → saveOrderResults(false) + return
이유: 결제 페이지로 넘어가지 않으면 이후 모든 단계 의미 없음
```

### N-4 수정 방향
```
주문번호 못 찾음 → success: true 유지하되, 에러 로그 추가
이유: 결제는 이미 됐으므로 성공이지만, 주문번호 누락은 별도 기록 필요
메시지: "결제 완료, 주문번호 수동 확인 필요"
```

### N-5 수정 방향
```
결제 로그 저장 실패 → 에러 로그로 기록 (createAutomationErrors)
이유: 결제는 됐으므로 주문 흐름은 유지, 하지만 실패 추적 필요
```

## 수정 정책 (확정)

### 결제 전 단계 — 모두 STOP

| 단계 | 정책 | 이유 |
|---|---|---|
| 로그인 실패 | STOP | 진행 불가 |
| 장바구니 비우기 후 잔여 상품 | STOP | 이전 상품과 같이 결제될 위험 |
| URL 이동 실패 | STOP | 상품 페이지 접근 불가 |
| 옵션 선택 실패 | STOP | 잘못된 옵션으로 주문 위험 |
| 수량 입력 실패 | STOP | 잘못된 수량으로 결제 위험 |
| 수량 검증 실패 | STOP | 입력한 수량과 실제 수량 불일치 |
| 가격 추출 실패 | STOP | 가격 비교 불가 |
| 가격 5,000원 초과 | STOP | 비정상 가격 |
| 장바구니 담기 실패 | STOP | 결제 수량 틀어짐 |
| 장바구니 검증 실패 | STOP | 기대 상품과 실제 불일치 |
| 배송지 입력 실패 | STOP | 배송 불가 |
| 주문하기 버튼 못 찾음 | STOP | 결제 페이지 진입 불가 |

### 결제 후 단계 — 결제는 이미 됐으므로 멈출 수 없음

| 단계 | 정책 | 이유 |
|---|---|---|
| 주문번호 못 찾음 | CONTINUE + 에러 로그 | 결제 취소 불가, 수동 확인 필요 |
| saveOrderResults 실패 | CONTINUE + 에러 로그 | 결제 취소 불가, 대행접수/출고 수동 처리 |
| createPaymentLogs 실패 | CONTINUE + 에러 로그 | 결제 금액 추적 누락, 수동 확인 |
| createNeedsManagerVerification 실패 | CONTINUE + 에러 로그 | 담당자 확인 건 누락, 수동 확인 |

### 예외: needsManagerVerification (결제 전)

```
옵션 없음, URL 없음, 디자인파일 없음 등
→ 해당 상품만 스킵 (SKIP) + 담당자확인필요 기록
→ 배치: 나머지 상품이 전부 성공하면 결제 진행
→ 개별: 다음 상품으로 진행
이유: 시스템 데이터 문제이지 자동화 버그가 아님. 멈추면 다른 정상 상품까지 지연.
```

### 장바구니 비우기 — 검증 기반

```
clearCart() 시도 (실패해도 OK)
→ 장바구니 상품 개수 확인
→ 0개 → 진행
→ 1개 이상 → STOP ("장바구니에 이전 상품 잔여: N개")
이유: 장바구니가 이미 비어있으면 "삭제할 상품 없음" 에러가 나므로,
      clearCart 실패 자체로 멈추면 정상 상황에서도 중단됨.
```

## 수정 우선순위

### Phase 1: 🔴 CRITICAL (결제 금액 틀어지는 문제) — ✅ 완료
1. **ALL-1**: 장바구니 비우기 실패 → STOP ✅ (6개 벤더)
2. **ALL-2**: 배치 상품 일부 실패 → 전체 그룹 STOP ✅ (네이버/쿠팡)
3. **주문하기 버튼 못 찾음** → STOP ✅ (네이버)

### Phase 2: 🟡 HIGH (데이터 누락 문제) — ✅ 완료
4. **ALL-3**: 주문번호 못 찾음 → success: true + 에러 로그 ✅ (네이버/쿠팡/냅킨/성원)
5. **ALL-4**: 결제 로그 실패 → createAutomationErrors 기록 ✅ (6개 벤더)
6. **ALL-5**: 담당자확인 저장 실패 → console.error ✅ (4개 벤더)
7. **finalOrderNumber 스코프 버그** → 수정 ✅ (쿠팡)

### Phase 3: 에러 메시지 품질 개선 — 미착수
8. 에러 메시지에 디버깅 정보 추가 (URL, 셀렉터, 기대값/실제값)
9. inferErrorCode 키워드 추론 → 명시적 에러 코드 전달
